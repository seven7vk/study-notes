## 关键设计决策

**1. 加一个 `manager` 层**
这是阿里巴巴 Java 开发手册里的建议，用来承接：
- 多个 DAO 的组合操作
- 外部接口的封装（防腐层）
- 可复用但又不属于单个 Service 的逻辑

避免 Service 越写越肥。

**2. `repository` 封装 `mapper`**
不直接在 Service 里调 Mapper，通过 Repository 做一层隔离。好处是：
- 换 ORM 框架时只改这一层
- SQL 逻辑不会散落在 Service 里

**3. DTO / VO / Entity 严格分离**
- `Entity`：数据库映射对象，不对外暴露
- `DTO`：Service 层传输对象
- `VO`：Controller 层返回给前端的视图对象

哪怕字段一样，也用 MapStruct 转换，不要图省事直接透传 Entity。

**4. 统一异常 + 统一返回**
这是外包团队最容易乱的地方，必须在脚手架阶段就定死：
```java
// 统一返回
Result<T> { code, message, data }

// 全局异常处理
@RestControllerAdvice
```

## 给你的额外建议

| 问题 | 建议 |
|---|---|
| 业务需求未定 | 先搭脚手架，不要过度设计，需求稳定后再考虑模块拆分 |
| 团队水平参差 | 写一份《项目开发规范.md》比架构设计更有价值 |
| 人员流动风险 | 强制 Code Review，关键逻辑必须有注释 |
| 未来可能微服务 | 按**业务模块**建包（如`order/`、`user/`），而非按技术层平铺，拆服务时成本更低 |

## 什么时候才考虑DDD？

当满足以下条件再引入：
- 业务需求**稳定且复杂**（比如金融、电商核心域）
- 团队有人真正理解领域建模
- 项目生命周期足够长，值得投入建模成本

外包项目短平快，**可落地 > 架构先进**，这才是最重要的判断标准。
